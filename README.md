# PulseGuard

A production-style **real-time fraud detection pipeline**: transactions stream
through RabbitMQ, get enriched with rolling behavioral features from a
Redis-backed feature store, are scored by an XGBoost classifier served over
ONNX Runtime inside Spring Boot, and fan out to a live WebSocket dashboard —
all measured end to end.

```
simulator ──▶ RabbitMQ ──▶ consumer ──▶ Redis feature store ──▶ XGBoost (ONNX)
 (replay)      (stream)      (4-8x)       (rolling aggregates)     (scoring)
                                                                       │
                                dashboard ◀── STOMP WebSocket ◀────────┘
```

---

## Highlights

- **RabbitMQ streaming pipeline** — a simulator replays 56,962 held-out
  transactions onto a durable queue at a configurable rate (tested at 500
  txn/s); a concurrent listener pool (4–8 consumers, prefetch 64) does the
  scoring.
- **Redis-backed online feature store** — per card, a 60-second sliding window
  (sorted set) plus lifetime aggregates (hash) maintained in a **single
  pipelined round trip** per transaction: window count, seconds since last
  transaction, and amount-to-average ratio.
- **Offline/online feature parity** — the training script computes the exact
  same rolling features (state read *before* the current transaction is folded
  in) while replaying the dataset in time order, so the model sees identical
  semantics in training and serving.
- **XGBoost via ONNX Runtime** — trained in Python with SMOTE to counter the
  577:1 class imbalance, exported to ONNX (ZipMap stripped, parity-checked
  against `predict_proba` at export time), and served in-process from Java at
  **~70 µs per inference** (p50).
- **Live WebSocket dashboard** — a single static page (STOMP + Chart.js from
  CDN) showing transactions, fraud alerts, live recall against ground-truth
  labels, throughput, and p50/p95/p99 latency.
- **Testcontainers integration test** — boots real Redis + RabbitMQ containers,
  pushes transactions through the full pipeline, and asserts on scores,
  feature-store state, and latency accounting.

## Model quality (held-out test split)

Trained on the [ULB credit-card fraud dataset](https://www.openml.org/d/1597)
(284,807 transactions, 492 frauds), 80/20 stratified split, SMOTE applied to
the training split only:

| metric | value |
|---|---|
| ROC-AUC | **0.978** |
| PR-AUC | 0.883 |
| Recall @ 0.5 | **0.867** |
| Precision @ 0.5 | 0.810 |

## Pipeline benchmark

Sustained on an Apple Silicon laptop (app + Redis + RabbitMQ all local),
measured by the pipeline itself over a 4,096-sample sliding reservoir:

| metric | p50 | p95 | p99 |
|---|---|---|---|
| End-to-end latency (publish → scored) | 2.5 ms | 4.2 ms | 6.8 ms |
| ONNX inference | 69 µs | 144 µs | 242 µs |
| Feature store round trip | 1.7 ms | 2.9 ms | 4.9 ms |

Throughput: **500 txn/s sustained** (simulator-limited, not pipeline-limited).

## Running it

Requires: Java 25, Maven, Docker, Python 3.11+ (training only).

**1. Train the model** (downloads the dataset from OpenML on first run,
writes `model/fraud_xgb.onnx` and `data/stream.csv.gz`, ~2 min):

```bash
python3 -m venv training/.venv
training/.venv/bin/pip install -r training/requirements.txt
training/.venv/bin/python training/train.py
```

**2. Start the infrastructure:**

```bash
docker compose up -d     # Redis :6379, RabbitMQ :5672 (management UI :15672, pulseguard/pulseguard)
```

**3. Run the app:**

```bash
mvn spring-boot:run
```

Open **http://localhost:8080** — the simulator autostarts at 50 txn/s. Crank
it up from the dashboard (or `curl -X POST 'localhost:8080/api/simulator/start?tps=500'`)
and watch latency percentiles hold single-digit milliseconds.

**4. Run the integration test** (spins up its own Redis + RabbitMQ):

```bash
mvn test
```

WebSocket (STOMP over `/ws`): `/topic/transactions` (sampled stream),
`/topic/frauds` (every flagged transaction), `/topic/stats` (1 Hz snapshots).

## How the features stay honest

The dataset's PCA features (V1–V28) carry most of the fraud signal, but the
point of the feature store is the *architecture*: real deployments score on
behavioral aggregates that must be computed identically offline and online.
PulseGuard enforces that discipline end to end:

1. Training replays the dataset in time order and computes each card's window
   count, time-since-last, and amount ratio **before** folding the current
   transaction into the card's state.
2. Serving does the same read-before-write against Redis, pipelined into one
   round trip.
3. The ONNX export is parity-checked against XGBoost's own `predict_proba`
   before it is written (max abs diff must be < 1e-3; observed 0.0).

Ground-truth labels ride along with each replayed transaction, which is what
lets the dashboard report **live recall** — the fraction of true frauds the
model flags as they stream through.
